Three-Layer Data Model

Layer 1: Product Catalog
Master template
Product name, description
Category, variants
Base price, cost price
Margin %
Available options (lining types, finishes)
Layer 2: Deal Line Item (CRM)
Customized instance
References catalog product
Width, drop (per window)
Room name, window position
Fixing, stack, lining, motor
Per-item notes & instructions
Qty, adjusted price
Layer 3: Order Line Item (OMS)
Inherited + fulfillment data
All specs from deal line item
Production status per item
Supplier/factory assignment
Quality check notes
Delivery scheduling
Product Management (standalone): Master catalog that defines what you sell. Manages pricing, variants, cost/margin. Feeds into both CRM and OMS. Accessible from its own sidebar item.
CRM → OMS (dependent): When deal reaches "Ordered", auto-creates order with all customized line items (specs, notes, dimensions). OMS inherits everything — no re-entry needed.
OMS → CRM (sync back): When order reaches "Delivered", deal auto-advances to "Fulfilled". Delivery confirmation date flows back to the deal.
Why Product Management is separate from OMS: Industry best practice (PIM pattern). The product catalog is configuration data that changes infrequently. Deal/order line items are transactional data with per-instance customization. Keeping them separate means: (1) pricing changes don't retroactively affect existing orders, (2) the catalog is accessible from any module, (3) cost/margin data stays in Product Management — CRM users see sell price only.
